# AI 深度研究助手 — 架构规划 v2

## Context

用户希望构建一个「输入模糊需求 → 输出专业研究报告」的 AI 工具。核心理念：用 AI 编排多个环节（对话澄清、搜索、验证、文档生成），培养系统级通用问题解决能力。

**UI 风格**：专业商务风，非 Mario 像素风。

---

## 核心设计原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **用户闸门** | 每个阶段产出必须经用户确认，才进入下一阶段（避免方向错误浪费 API 费用） |
| 2 | **成本可预期** | 每阶段开始前预估消耗（搜索次数、AI token），用户可选择继续/缩减/中止 |
| 3 | **"不知道"优于"瞎编"** | 找不到可靠来源时，明确标注而非强行生成 |
| 4 | **渐进深度** | 先广度扫描再深度挖掘，支持用户说「这个地方再深入」 |
| 5 | **单一输出优先** | v1 只做一种报告类型做好，v2 再扩展 |

## 四阶段流水线（强化版）

```
用户输入一句话需求
      │
      ▼
┌─────────────────────────────────────────┐
│ 阶段 1：需求访谈 → 研究纲要              │
│                                         │
│ 1a. 结构化提问（4 个维度自动追问）        │
│     - 目的: 你要这份报告做什么决定？       │
│     - 受众: 给谁看？专业度多深？          │
│     - 边界: 什么不需要？地域/时间范围？    │
│     - 形式: 报告/指南/对比分析？          │
│                                         │
│ 1b. 生成《研究纲要》                     │
│     - 核心问题（1 句话）                  │
│     - 子问题拆解（3-5 个）               │
│     - 搜索关键词（每子问题 2-3 组）       │
│     - 预期信源类型                       │
│     - 预估成本                            │
│                                         │
│ 1c. ⏸ 用户闸门：确认纲要后再搜索         │
└──────────┬──────────────────────────────┘
           │ ✅ 用户确认
           ▼
┌─────────────────────────────────────────┐
│ 阶段 2：多源搜索 → 采编素材库             │
│                                         │
│ 2a. 查询生成                             │
│     - 每个子问题 → 3 组搜索词（中文+英文） │
│     - 每组搜索词 → 实际搜索（取前 5 条）   │
│                                         │
│ 2b. 并行搜索 + 采集                      │
│     - 每个搜索结果 → 抓取全文/摘要        │
│     - 自动标注：URL、发布时间、域名        │
│                                         │
│ 2c. 去重 + 时效性标注 + 信源分级          │
│     - 时效标记：🟢新 / 🟡偏旧 / 🔴过时    │
│     - 信源分级：A官方学术 / B知名媒体 / C个人 │
│                                         │
│ 2d. ⏸ 用户闸门：展示素材摘要（来源数、时效分布、信源质量），确认后核查  │
└──────────┬──────────────────────────────┘
           │ ✅ 用户确认
           ▼
┌─────────────────────────────────────────┐
│ 阶段 3：事实核查 → 验证素材               │
│                                         │
│ 3a. AI 识别关键事实主张                   │
│     - 从素材中提取"可被验证的断言"         │
│     - 跳过观点、推测、个人经验              │
│                                         │
│ 3b. 对每条关键事实交叉验证                 │
│     - 至少 1 个独立来源确认               │
│     - 无法验证 → 标注「⚠ 单一来源」        │
│     - 来源冲突 → 标注「⚠ 存在争议」并列出双方 │
│                                         │
│ 3c. 生成《验证报告》                      │
│     - 已验证事实（来源列表）               │
│     - 单一来源事实（需注意）               │
│     - 无法验证的声明（建议删除或标注）      │
│                                         │
│ 3d. ⏸ 用户闸门：确认验证结果后再生成文档  │
└──────────┬──────────────────────────────┘
           │ ✅ 用户确认
           ▼
┌─────────────────────────────────────────┐
│ 阶段 4：文档生成 → PPTX / DOCX            │
│                                         │
│ 4a. 生成文档大纲（用户确认后再写全文）     │
│                                         │
│ 4b. 按模板逐节生成内容                    │
│     - 每条内容自动关联来源脚注             │
│     - 时效性警告自动附在过时信息旁          │
│                                         │
│ 4c. 输出 PPTX 或 DOCX                     │
│     - 可下载、可编辑                      │
│     - 尾页：完整参考文献列表 + 方法说明    │
└─────────────────────────────────────────┘
```

## 输出类型（v1 只做 2 种）

| 类型 | 适用场景 | 文档结构 |
|------|---------|---------|
| **深度报告** | 分析性/学术性主题 | 摘要→背景→方法论→发现→结论→参考文献 |
| **执行指南** | 有具体困境，要可操作步骤 | 问题定义→方案对比→推荐方案→分步执行→资源清单 |

v2 再扩展：市场调研、趋势预测。

## 信息源处理规则

### 时效性标注

| 标记 | 技术文档 | 市场数据 | 新闻事件 |
|------|---------|---------|---------|
| 🟢 有效 | <2 年 | <1 年 | <6 月 |
| 🟡 偏旧 | 2-4 年 | 1-2 年 | 6-12 月 |
| 🔴 过时 | >4 年 | >2 年 | >12 月 |

过时内容不自动排除（历史分析可能需要），但需在前端明显标注。

### 信源分级

| 级别 | 来源 | 处理方式 |
|------|------|---------|
| A | 政府/官方文档/学术期刊 | 优先引用 |
| B | 知名媒体/咨询公司/行业报告 | 正常引用 |
| C | 个人博客/论坛/知乎/公众号 | 可引用但标注 |

## 技术选型

| 层 | 选择 | 理由 |
|----|------|------|
| 后端 | Node.js + Express | 复用抖音项目经验 |
| AI | DeepSeek API（接口抽象化） | 未来可切换模型 |
| 搜索 | **先用 WebSearch 内置工具，后续接 Bing/SerpAPI** | v1 零成本启动 |
| 数据库 | SQLite（better-sqlite3） | 研究历史持久化，零配置 |
| 文档 | python-pptx + python-docx | 真正的 Office 格式 |
| 前端 | 原生 HTML/CSS/JS | 零框架，**专业商务风** |
| 异步 | 内存队列 | 先简单，后可换 Redis |

## 渐进里程碑

### M1：项目脚手架 + 阶段 1 简陋版（1-2 天）
- Express 空壳 + 前端骨架 + SQLite schema
- 一段硬编码的"访谈对话"直接输出研究纲要
- **可验证**：浏览器打开 → 看到页面 → 输入一个需求 → 输出纲要

### M2：阶段 1 完整对话引擎（1-2 天）
- 4 维度自动追问（目的/受众/边界/形式）
- 研究纲要自动生成
- 用户确认闸门
- **可验证**：和一个模糊需求对话 3-4 轮，输出完整纲要

### M3：阶段 2 搜索引擎（2-3 天）
- 纲要 → 搜索词拆解 → 并行搜索
- 时效性 + 信源标注
- **可验证**：给定纲要，返回 10-30 条带标注的素材

### M4：阶段 3 事实核查（1-2 天）
- AI 识别关键事实主张
- 交叉验证 + 标注
- **可验证**：给定素材，输出验证报告

### M5：阶段 4 文档生成（2-3 天）
- python-pptx + python-docx 报告生成
- 大纲→逐节生成→来源脚注
- **可验证**：给定验证素材，下载 PPTX/DOCX
- ✅ 已完成（支持 DOCX + PPTX，内容智能分页）

### M6：全流程打通 + 历史记录（1-2 天）
- 端到端：需求 → 报告
- SQLite 存储每次研究记录
- **可验证**：完整的从需求到下载报告的闭环
- ✅ 已完成（JSON 文件持久化 + 会话恢复 + 删除管理）

## 强化点总结（相比 v1 框架）

| 原框架 | 强化后 |
|--------|--------|
| "多轮对话澄清" | 4 维度结构化访谈 + 研究纲要正式文档 |
| 无用户闸门 | 每阶段结束 → 用户确认 → 下一阶段 |
| 搜索策略模糊 | 子问题→搜索词组→并行搜索→去重→标注 |
| 全量交叉验证（不现实） | 只验证 AI 识别的关键事实主张 |
| 4 种输出类型 | v1 只做 2 种（深度报告 + 执行指南），做好再扩展 |
| 无成本预估 | 每阶段开始前给估算 |
| 信源分级无指导 | A/B/C 三级 + 具体处理方式 |
| UI 风格未定 | **专业商务风**（确认） |
| 计划文件混乱 | 每个项目独立 PLAN.md，不共用 |
